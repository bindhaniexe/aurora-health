## Project Overview

We are building **Aurora** — a mobile-first AI health companion app using Expo.

Aurora helps users understand their health data through:

- Hydration tracking with an animated virtual water bottle
- Sleep tracking and weekly trend analysis
- Habit creation, completion, and streak tracking
- A voice-first AI companion (OpenAI Realtime API) that logs data through conversation
- Personalized daily insight cards generated by Groq AI
- A beautiful dark, premium, calm UI

This is a hackathon MVP. The goal is to build a polished, functional prototype that demonstrates product thinking, AI integration, and high-quality mobile UI — not a production-ready application.

## Architecture Guidelines

Use this structure unless there is a strong reason to change it:

```txt
app/
  (auth)/
  (onboarding)/
  (tabs)/
components/
constants/
hooks/
lib/
services/
stores/
types/
utils/
assets/
supabase/
```

### app/

Use for routes and screens only.

Screens should compose components and call hooks/stores. They should not contain large reusable UI blocks or complex business logic.

### components/

Create a component only when:

- It is reused in multiple places
- It makes a screen significantly easier to read
- It represents a clear UI concept: `WaterBottle`, `HydrationCard`, `InsightBanner`, `MicButton`, `ProgressRing`, `HabitItem`

Do not create tiny one-off components too early. When unsure, ask:

> Should this UI be extracted into a reusable component, or should I keep it inside the screen for now?

### services/

All Supabase database access goes here. Screens and stores never call Supabase directly.

```txt
services/
  hydrationService.ts
  sleepService.ts
  habitService.ts
  insightService.ts
  agentService.ts       ← executes voice agent function calls
  profileService.ts
```

### stores/

Zustand stores for global client state.

```txt
stores/
  authStore.ts
  profileStore.ts
  hydrationStore.ts
  sleepStore.ts
  habitStore.ts
  companionStore.ts
```

### lib/

External service helpers and clients.

```txt
lib/
  supabase.ts           ← Supabase client init
  groq.ts               ← Groq client
  realtimeAgent.ts      ← OpenAI Realtime API connection manager
  cn.ts                 ← NativeWind class merge utility
```

### hooks/

Custom hooks that compose store + service logic.

```txt
hooks/
  useHydration.ts
  useSleep.ts
  useHabits.ts
  useHealthSummary.ts   ← aggregates all data for AI context
```

### constants/

Design tokens, images, and static config.

```txt
constants/
  colors.ts
  typography.ts
  spacing.ts
  images.ts             ← centralized image imports (see Image Rule below)
  agentTools.ts         ← OpenAI Realtime function definitions
```

### types/

All TypeScript interfaces in one place.

```txt
types/
  index.ts
```

### utils/

Pure utility functions.

```txt
utils/
  dateUtils.ts
  healthUtils.ts        ← goal calculations, streak logic, hydration percentages
  formatUtils.ts
```

## Style Exception Rules

Use `StyleSheet` or inline styles for these React Native components instead of NativeWind:

| Component / Scenario | Why | Use Instead |
|---|---|---|
| **SafeAreaView** | `className` not supported | Inline styles or `StyleSheet` |
| **Button** | Only `title` and `onPress` props | Use `TouchableOpacity` with custom styles |
| **KeyboardAvoidingView** | Behavior props unsupported | Inline styles or `StyleSheet` |
| **Modal** | `visible`, `transparent` props | Inline styles |
| **ScrollView** | `contentContainerStyle`, `indicatorStyle` | `StyleSheet` |
| **TextInput** | Input-specific props | Inline styles |
| **Animated.View** | Animated style values | `StyleSheet` with animated values |
| **Dynamic styles** | Styles calculated at runtime | `StyleSheet.create()` or inline |
| **Platform-specific** | iOS/Android-only props | Conditional inline styles |
| **Pressable / TouchableOpacity** | Pressed state styles | `StyleSheet` |
| **Shadow (iOS/Android)** | Different shadow syntax per platform | `StyleSheet` with platform checks |
| **Transform arrays** | Complex transform combinations | `StyleSheet` |
| **SVG / Reanimated** | Custom animated and SVG props | `StyleSheet` or inline |

### SafeAreaView Example

```tsx
// ✅ CORRECT
import { SafeAreaView } from "react-native-safe-area-context";
function MyScreen() {
  return (
    <SafeAreaView style={{ flex: 1, backgroundColor: colors.bgPrimary }}>
      {/* content */}
    </SafeAreaView>
  );
}

// ❌ INCORRECT
function MyScreen() {
  return <SafeAreaView className="flex-1 bg-[#0A0E1A]">{/* content */}</SafeAreaView>;
}
```

Otherwise, always use NativeWind utilities.

## State Management Rules

Use **Zustand** for global client state.

Use **local `useState`** for temporary UI state (modals open, input values, loading spinners).

Persist to **Supabase** for all health data (hydration logs, sleep logs, habits, completions).

Use **AsyncStorage** only for lightweight local preferences (units, onboarding flags).

### Optimistic Updates Pattern

Always apply optimistic updates for data mutations:

```ts
addWater: async (amount_ml: number) => {
  // 1. Update UI immediately
  set(state => ({ todayTotal: state.todayTotal + amount_ml }));
  try {
    // 2. Persist to Supabase
    const log = await hydrationService.addLog(amount_ml);
    set(state => ({ todayLogs: [log, ...state.todayLogs] }));
  } catch {
    // 3. Rollback on failure
    set(state => ({ todayTotal: state.todayTotal - amount_ml }));
  }
}
```

## Supabase Rules

All database access goes through `services/` files. Screens and stores never call Supabase directly.

Every table has **Row Level Security** enabled. Users can only read and write their own data.

The service layer pattern:

```ts
// services/hydrationService.ts
export const hydrationService = {
  async getTodayLogs(): Promise<HydrationLog[]> { ... },
  async addLog(amount_ml: number): Promise<HydrationLog> { ... },
  async getWeeklyLogs(): Promise<HydrationLog[]> { ... },
};
```

Never expose Supabase service role keys in the mobile app. Use only the anon key on the client.

## OpenAI Realtime API Rules (Voice AI)

The voice companion is the most important feature. Treat it as the centerpiece of the app.

### Connection Manager (`lib/realtimeAgent.ts`)

- Manages the WebSocket/WebRTC connection to `gpt-4o-realtime-preview`
- Injects user health summary as the system prompt on session start
- Handles incoming events: `response.audio.delta`, `response.function_call`, `response.text.delta`
- Executes agent function calls via `agentService.ts`, then sends `tool_result` back to the session

### Agent Function Definitions (`constants/agentTools.ts`)

Five functions the AI can call:
- `log_water(amount_ml)` — logs water intake
- `log_sleep(hours, quality?)` — logs sleep
- `create_habit(name, frequency)` — creates a new habit
- `complete_habit(habit_name)` — marks a habit done today
- `get_health_summary()` — fetches recent data so the AI can answer "how am I doing" questions

### Security

The OpenAI API key must **never** appear in client code.

Use a **Supabase Edge Function** to create an ephemeral Realtime session token and return it to the app:

```
Client → Supabase Edge Function (authenticated) → OpenAI /v1/realtime/sessions → returns client_secret → Client uses token
```

### System Prompt

Build the system prompt dynamically from the user's profile and recent health data:

```ts
function buildSystemPrompt(profile: Profile, summary: HealthSummary): string {
  return `
You are Aurora, a warm and encouraging personal health companion.
The user's name is ${profile.name}. Their goals: ${profile.goals.join(', ')}.
Today: ${summary.todayWaterMl}ml water of ${profile.water_goal_ml}ml goal.
Sleep last night: ${summary.lastSleepHours}h (goal: ${profile.sleep_goal_hrs}h).
Habits done: ${summary.habitsCompleted}/${summary.habitsTotal}.
Memory: ${profile.memory_notes || 'None yet.'}.
Use your tools when the user mentions data — don't ask, just act and confirm.
Be concise. Max 2 sentences per response. Never be clinical.
  `.trim();
}
```

## Groq Rules (Insight Generation)

Use Groq (`llama-3.3-70b-versatile`) for generating the daily insight card text on the dashboard.

- Generate once per day, cache result in Supabase `insights_cache` table
- Check cache before calling Groq; skip generation if a valid insight exists for today
- Keep prompt concise — request one sentence, max 20 words
- Max tokens: 60

The Groq API key can be set as an `EXPO_PUBLIC_` env variable for the client in the hackathon build. For production, proxy through a Supabase Edge Function.
